Anthropic context:
We are targeting the Anthropic sponsor prize track. Their criteria is: projects built with Claude Code that tackle meaningful issues in health, education, economic opportunity, or any domain where AI could genuinely shift what's possible for people. The prize is $5,000 in API credits, an office hour with their Applied AI team, and an invitation to visit their SF office.
In LumaLens, Anthropic is the core intelligence layer. Claude Vision (claude-sonnet-4-6) receives up to 5 exterior photos of a property scraped from Zillow and returns structured JSON describing the house geometry — number of stories, roofline shape (flat/gabled/hipped), facade width estimate in feet, primary material (brick/stucco/wood/stone), and notable architectural features like columns, garage doors, porches, or dormers. This JSON is what drives the entire Three.js procedural model. Without Claude, there is no 3D scene.
The meaningful impact angle: landscape lighting installers currently have to set up physical temporary lighting rigs on a property — renting equipment, sending a crew, spending hours on site — just to show a client what the finished install might look like, with no guarantee the client approves and pays. LumaLens replaces that physical demo with a digital preview generated from publicly available listing photos, saving installers significant time and cost on every sales call. Claude is what makes the property-specific geometry extraction possible without requiring expensive photogrammetry or LiDAR hardware.
Architecture decisions made specifically for this track: we use Claude Vision rather than a generic depth estimation model because it allows us to extract semantically meaningful geometry (roofline type, material, features) that maps directly to lighting placement logic. We cap photos at 5 to keep latency under 10 seconds for a live demo. The analyze API route at app/api/analyze/route.ts should use the Anthropic SDK, send all photos as base64 image blocks in a single message, and return a strictly typed GeometryJSON object. Claude should be prompted to respond only in JSON with no preamble.

Browserbase context:
We are targeting the Browserbase sponsor prize track. Their criteria requires participants to build an agent that uses the web, powered by the Browserbase platform, using one of: browsers, search, fetch, Stagehand, or Browse CLI.
In LumaLens, Browserbase handles the scraping layer. When a user pastes a Zillow listing URL, our Next.js API route at app/api/scrape/route.ts spins up a Browserbase browser session, navigates to the listing, waits for the photo gallery to load, and extracts all exterior photo URLs from the image carousel. It returns a JSON array of up to 5 photo URLs which are then passed to the Claude Vision analyze route.
Architecture decisions made specifically for this track: we use Browserbase rather than a simple fetch or cheerio scrape because Zillow heavily relies on JavaScript rendering and actively blocks headless requests — a real browser session is required. This is the correct technical justification for using Browserbase and should be mentioned explicitly in the Devpost and in the judge pitch. The scrape route should handle the case where Zillow's selectors have changed by implementing 2-3 fallback selector strategies for finding photo URLs. If all selectors fail, the route should return a descriptive error that triggers the sample house fallback in the UI rather than crashing.
The Browserbase SDK should be initialized with BROWSERBASE_API_KEY from environment variables. The session should be closed and cleaned up after photo extraction regardless of success or failure to avoid leaving open sessions.
